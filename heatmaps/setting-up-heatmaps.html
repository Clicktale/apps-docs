Setting up Heatmaps

Setting up Heatmaps

In Clicktale for Apps, heatmap functionality comes out of the box as the SDK will capture all the taps and other gestures that your users are performing in the application.

We later take that data, and group it according to ‘Screens’ or ‘Page Views’. This is done to logically capture the correct data per screen, or data layer that you display in your application, so for example:

  1. The application starts with a ‘Home Page’, showing some highlights of your products.
  2. A user clicks on a Menu, the menu layer appears on the screen
  3. A user then either clicks on some item to view the next screen, or goes back to the ‘Home Page’

So the ‘Home Page’ screen was displayed, and interactions happened on it, and then later a Menu screen, or layer was displayed.

We can show you the interactions aggregated only for the ‘Home Page’ and then only for the menu layer – keep your analysis accurate and conclusions to the point.

All that is required is signaling to the SDK which logical screen is currently showing – through the API. More information on how to do so is available at the using events article of the specific platform you are working on: iOS or Android.

Out of the box, we monitor ViewController initializations on iOS and Activities on Android and send an automatic Screen View event as usually these would signify a logical change in the application. This behavior can be controlled via configuration and be turned off if it does not match the logic of your app.

For more complex applications, it is important to remember to fire the Screen/Page View event not only at the creation of the current view, but also when transitioning.

So in the example above, after the user looked at the menu layer, if he then went back to the 'Home Page', you need to fire that event again so the SDK knows the menu visuals are not longer displayed, and the taps collected right now are relevant for the home screen again.
